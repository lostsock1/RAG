## Product goal

Build an API-first, ACL-aware RAG platform that reliably indexes and answers from both textbooks and loose documents. It must support small corpora and very large corpora, with strong citations, negative-answer behavior, and evaluation.

## Current architecture baseline

- One platform, two document profiles:
  - Book profile
  - Loose document profile
- Shared search and answer core:
  - BM25 / phrase / exact search
  - BGE-M3 dense search
  - BGE-M3 sparse search
  - Fusion
  - Parent-child expansion
  - BGE reranker
  - Context builder
  - LLM answer
  - Sentence-level evidence verifier
- Web UI is a client of the public API.
- Backend owns security and ACL enforcement.

## Open risks

- **LLM answers leak internal context labels (`rank=N`) to users** — surfaced by the Phase D grounding measurement (incl. a garbled `rank=!!!…2` artifact). Source: `llm_backend.py:204` renders `rank={block.rank}` into prompt block headers and the system instruction does not forbid echoing them. Fix: human-oriented source labels + instruction rule (early Phase E); re-measure ADR-0019 criterion 1 after.
- **Qdrant corpora indexed before 2026-06-10 (incl. the VPS) return no dense/sparse results until re-ingested** — the A5 expiry filter is fail-closed on the new `expires_at_ts` payload field. Re-ingest the VPS corpus (or wait for the E4 reindex CLI) before exercising search there.
- ~~ADR-0017 streaming SLA failing by design~~ **Resolved 2026-06-10 by ADR-0018 + verification gate** (P50 3.11s / P95 3.22s; load test green). Residual: the ADR-0008 ~2s ambition is provider-bound until local LLM serving (master plan Phase G).
- Exact target corpus size not yet measured.
- No local GPU hardware — all LLM calls go through ppq.ai API (acceptable for testing, must be resolved before air-gapped production).
- Llama 3.3 multilingual *generation* quality still unverified on German/Portuguese — Phase D/E5. (Retrieval quality on DE/PT is now measured: BGE-M3 scores 1.000 recall@10/nDCG@10/MRR@10 on the C5 subset, n=7 each — retrieval side de-risked 2026-06-11.)
- OCR quality requirements not validated on production corpus.
- ACL model not mapped to real organization roles/groups.
- ppq.ai is a proxy intermediary — direct provider fallback (Together, Groq) needed if ppq.ai has availability issues.
- Browser-level frontend verification against the running VPS API is still unverified.
- The VPS `.env` uses `OIDC_SCOPES_CLAIM=permissions`; local tests rely on the default `scope` claim path. Test isolation should be tightened so environment-specific claim mapping does not skew remote test expectations.
- **VPS .venv was missing the ingestion-extras deps** (`opensearch-py`, `qdrant-client`, `temporalio`) — installed during the 2026-05-23 deploy. The heavier ML deps (`torch`, `FlagEmbedding`, `sentence-transformers`) were NOT installed because (a) the VPS network is flaky and pip downloads of multi-hundred-MB wheels kept resetting, (b) the 12-point check does not exercise the embedder/reranker. Anyone running the full ingestion → search → answer path on the VPS must `pip install -e .[ingestion]` first (and may need to retry on a more stable network or split the install into smaller groups).
- **VPS Postgres backup retained** at `~/backups/uber_rag-pre-audit-20260523-145247Z.sql` (648 lines, taken before the 8-migration upgrade). Safe to delete after a week if no follow-up incident.
- **The two migration bugs (0006 boolean default, 0007 sa.table reflection) escaped CI** because both surfaced only on Postgres and CI runs SQLite. Add `tests/integration/test_migrations.py` cases that exercise the full chain against a Postgres test container before claiming Postgres-clean for any future migration.
